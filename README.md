## DeepSeek API

LibreLog now uses the DeepSeek API instead of loading a local HuggingFace model.
Secrets are read from `.env` or process environment variables. Copy `.env.example` to `.env` locally and fill values before running:

```powershell
Copy-Item .env.example .env
```

Required/common variables:

```text
DEEPSEEK_API_KEY=
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-flash
NEO4J_URI=bolt://localhost:7687
NEO4J_USER=neo4j
NEO4J_PASSWORD=
NEO4J_DATABASE=neo4j
```

`.env` is ignored by git. Do not commit real API keys or database passwords.
You can still override LLM settings from the command line with `--api_key`, `--model`, and `--api_base`.

## Parsing

Please run the following command to run LibreLog.
```powershell
python evaluation.py --project all --sample 3 --write_group_tree
```

The parser now runs through the agent runtime in `agent/runner.py`. It reads raw `.log` files under `full_dataset/` and automatically skips `-label.log` files.
Before parsing, it extracts recognized timestamp fields from each raw log line and writes the result to `preprocessed.csv` beside the parsing outputs.
You can also target a subset, for example:

```powershell
python evaluation.py --project "auth,dnsmasq.log,intranet_server" --model "deepseek-v4-flash" --sample 3
```

Results are written to `result_deepseek/`. Add `--write_group_tree` to persist `group_tree.json`; Jaccard/cosine similarity, representative sampling, regex memory lookup, regex validation, POI/relation schema preparation, and reflection are exposed as agent tools under `agent/tools/`.
For each log source, the agent now prepares `poi_schema.csv`, `relation_schema.csv`, and `schema_meta.json` beside the parsing outputs. Existing schemas are loaded from `schemas/`; missing schemas are generated by DeepSeek and aligned against existing canonical POI fields and relations.
DeepSeek also plans the source-level tool schedule. Use `--disable_planner` if you need the deterministic default plan.

For a local smoke test without calling DeepSeek:

```powershell
python evaluation.py --input_dir .tmp_agent_dataset --output_dir .tmp_agent_result --mock_llm --write_group_tree
```

## Frontend Viewer

Start the integrated parser + knowledge-graph workbench with:

```powershell
python -m frontend.server --port 8765
```

Open `http://127.0.0.1:8765` to run the full agent pipeline. The backend exposes JSON/SSE APIs and the static frontend calls those APIs, so the UI remains front/back separated.

The workbench now covers both projects in one flow:

1. LibreLog reads raw `.log` files from `full_dataset/` and writes parser artifacts to `result_deepseek/`.
2. The backend mirrors each parser `3.csv` and matching schema files into `edc-log/AIT/` and `edc-log/schemas/`.
3. `edc-log/log_pipeline_agent` runs preflight analysis, DAG planning, parameter extraction, per-source graph building, fused graph generation, optional Neo4j writing, and graph queries.

The frontend keeps the original parser views for raw input, `preprocessed.csv`, `group.csv`, parser output CSV files, optional `group_tree.json`, POI schemas, relation schemas, summaries, and agent traces. It also keeps the edc-log views for dataset selection, preflight output, DAG plan, live execution timeline, graph summary, artifact browsing, local graph query, Neo4j query, and Neo4j cleanup.
